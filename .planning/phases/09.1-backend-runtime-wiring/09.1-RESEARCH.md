# Phase 9.1: Backend Runtime Wiring - Research

**Researched:** 2026-04-11
**Domain:** Python backend — SimulationEngine wiring, Event lifecycle integration, Agent OOP delegation
**Confidence:** HIGH

---

## Summary

Phase 9.1 is a gap-closure phase: all the pieces exist and are unit-tested, but three integration points were never wired in Phase 7. The `Event` schema class in `backend/schemas/events.py` has a complete lifecycle state machine (created/active/spreading/expired), 17 passing unit tests, and `is_expired()` / `tick()` methods — but no production code ever instantiates an `Event` object. The `Agent` class has `perceive()`, `decide()`, `converse()`, and `reflect()` wrapper methods — but `engine._agent_step()` calls `perceive()` and `decide_action()` directly from imported modules, bypassing two of the three live wrappers.

The work is three targeted code edits in `backend/simulation/engine.py`, each independent of the others. The edits are small in character count but critical for requirement closure: (1) `inject_event()` must create an `Event` object and store it in a new `self._active_events` dict before calling `add_memory()`; (2) `_agent_step()` must call `agent.perceive(maze, all_agents_view)` instead of `perceive(...)` directly; (3) `_agent_step()` must call `agent.decide(simulation_id, perception, ...)` instead of `decide_action(...)` directly; (4) the tick loop must advance event lifecycle and expire old events. No behavioral change is expected for users — the event still reaches ChromaDB via `add_memory()`, agents still perceive and decide the same way.

**Primary recommendation:** Three targeted surgical edits to `engine.py`, new `self._active_events: dict[str, Event]` storage, and new integration tests covering runtime Event instantiation, `heard_by` population, expiry enforcement, and `_agent_step` wrapper routing.

### Pattern 1: Event Object Registry in Engine

**What:** `SimulationEngine` maintains `self._active_events: dict[str, Event]` keyed by a stable ID (UUID or `f"{mode}:{text[:50]}:{created_tick}"`). Events are added by `inject_event()`, advanced each tick by `_tick_loop()`, and excluded from agent context when `is_expired()` returns True.

**When to use:** Any time an event is injected into the simulation.

**Example:**
```python
# Source: backend/schemas/events.py (verified)
from backend.schemas.events import Event, EVENT_EXPIRY_TICKS
import uuid

# In inject_event():
event_id = str(uuid.uuid4())
event = Event(
    text=text[:500],
    mode=mode,
    target=target,
    status="created",
    created_tick=self._tick_count,
    expires_after_ticks=EVENT_EXPIRY_TICKS,
)
event.tick(self._tick_count)          # transitions created -> active
self._active_events[event_id] = event
# ... then add_memory() as before

# In _tick_loop(), after agent steps:
for eid, ev in list(self._active_events.items()):
    ev.tick(self._tick_count)
    if ev.is_expired(self._tick_count):
        del self._active_events[eid]
```

### Pattern 2: Agent Wrapper Routing in _agent_step

**What:** Replace the two direct module function calls with Agent method calls. The `Agent.perceive()` and `Agent.decide()` wrappers have the same signatures as the underlying functions (they take the same parameters and return the same types), so this is a mechanical substitution.

**When to use:** Every `_agent_step()` invocation.

**Example:**
```python
# Source: backend/agents/agent.py (verified — Agent.perceive signature)
# BEFORE:
perception = perceive(
    agent_coord=agent.coord,
    agent_name=agent_name,
    maze=self.maze,
    all_agents=all_agents_view,
)

# AFTER:

# BEFORE:
action = await decide_action(
    simulation_id=self.simulation_id,
    agent_name=agent_name,
    agent_scratch=config.scratch,
    agent_spatial=config.spatial,
    current_activity=agent.current_activity,
    perception=perception,
    current_schedule=agent.schedule,
    open_locations=open_locs,
    last_sector=agent.last_sector,
    new_perceptions=bool(perception.nearby_agents or perception.nearby_events),
    schedule_changed=schedule_changed,
)

# AFTER:
action = await agent.decide(
    simulation_id=self.simulation_id,
    perception=perception,
    open_locations=open_locs,
    last_sector=agent.last_sector,

# Agent.decide() passes **kwargs to decide_action — open_locations, last_sector,
```

### Pattern 3: heard_by Population on Perception

**What:** After `agent.perceive()` returns, iterate `self._active_events` and append `agent_name` to `event.heard_by` for events that are not expired and not already in `heard_by`. This tracks which agents perceived which events.

**Note on broadcast events:** Per design decision D-09 in `Event.__doc__` ("Broadcasts do not track propagation — heard_by always empty"), `heard_by` population should only apply to whisper-mode events. Broadcast events have `heard_by = []` by design.

**Example:**
```python
# After perception in _agent_step():
for ev in self._active_events.values():
    if ev.mode == "whisper" and not ev.is_expired(self._tick_count):
        if agent_name not in ev.heard_by:
            ev.heard_by.append(agent_name)
```

### Anti-Patterns to Avoid

- **Removing `add_memory()` from `inject_event()`:** Event objects are the lifecycle tracking layer ON TOP of ChromaDB. The existing `add_memory()` call must stay — agents read events through memory retrieval. The `Event` object is a separate runtime record, not a replacement.
- **Storing events in Maze tiles:** The Maze tile `_events` dict is for map-level events (from tile data), not for user-injected events. Injected events should live in `self._active_events` on the engine.
- **Modifying `Event` as a mutable shared object without care:** `Event` is a Pydantic `BaseModel`, not a `dataclass`. Pydantic v2 models are mutable when `model_config = ConfigDict(frozen=False)` is the default, so `.heard_by.append()` works — but avoid reassigning the model (use mutation, not replacement).
- **Moving the `Event` import inside the method:** Import `Event` at the module level in `engine.py` to keep it visible and auditable. Inline imports (the pattern used in `agent.py`) were chosen to break circular imports — `engine.py` has no such constraint.
- **Breaking the test for `inject_event`:** The existing `test_event_injection.py` tests mock `add_memory` at the `backend.simulation.engine.add_memory` patch path. After adding `Event` creation, `add_memory` still must be called — these tests will continue to pass without changes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Event ID generation | Custom string key logic | `uuid.uuid4()` | UUID guarantees uniqueness across concurrent injections; string slicing creates collisions |
| Event lifecycle state machine | Manual if/else status transitions | `Event.tick()` (already exists) | 17 unit tests validate it; don't duplicate |
| Event expiry check | `current_tick - created_tick >= N` inline | `Event.is_expired()` (already exists) | Method exists, tested, correct |
| Agent perception routing | Direct `perceive()` call | `agent.perceive()` (already exists) | Wrapper exists — use it |
| Agent decision routing | Direct `decide_action()` call | `agent.decide()` (already exists) | Wrapper exists, `**kwargs` forwards all engine-computed params |

**Key insight:** This phase is almost entirely "wire what already exists." The schema objects have correct behavior. The agent wrappers have correct delegation. The engine just bypasses them.

## Common Pitfalls

### Pitfall 1: Circular Import When Adding `Event` Import to engine.py

**What goes wrong:** Adding `from backend.schemas.events import Event` at the module level of `engine.py` could create a circular import if the Event schema imports anything from the simulation package.

**Why it happens:** The schema package may have layered imports.

**How to avoid:** Check the import chain before writing. `backend/schemas/events.py` only imports from `pydantic` and `typing` — no simulation imports. The module-level import is safe. [VERIFIED: read backend/schemas/events.py]

**Warning signs:** `ImportError: cannot import name 'Event' from partially initialized module`.

---

### Pitfall 2: `heard_by` on Broadcast Events — Design Decision Conflict

**What goes wrong:** The success criterion says "When an agent perceives an event, `Event.heard_by` is updated" but the Event schema docstring says "Broadcasts do not track propagation (heard_by always empty) — D-09."

**Why it happens:** The success criteria were written at a higher level than the schema's design decision.

**How to avoid:** Implement `heard_by` population for whisper events only, consistent with D-09. The success criterion says "an agent perceives an event" — for broadcast events, agents learn through `add_memory()` which already handles all agents. The `heard_by` field tracks whisper propagation (who has the gossip), not broadcast acknowledgment.

**Warning signs:** A test expecting `broadcast_event.heard_by == []` failing after wiring.

---

### Pitfall 3: Mutable Default for `self._active_events`

**What goes wrong:** Declaring `self._active_events = {}` in `__init__` is correct. Using a mutable default argument anywhere in the call chain is wrong.

**Why it happens:** Python mutable default argument bug — `def foo(events={})` shares the dict across all calls.

**How to avoid:** Always initialize in `__init__`: `self._active_events: dict[str, Event] = {}`. This is already the pattern used for `self._active_conversations`. [VERIFIED: engine.py line 108]

---

### Pitfall 4: `_agent_step` Refactor Breaking Existing Tests

**What goes wrong:** Changing `perceive(agent_coord=..., agent_name=..., maze=..., all_agents=...)` to `agent.perceive(maze=..., all_agents=...)` changes what patches need to intercept in tests.

**Why it happens:** Tests that mock `backend.simulation.engine.perceive` (the module-level import) will pass regardless — once `agent.perceive()` is called, it does a lazy import `from backend.agents.cognition.perceive import perceive as _perceive` and the mock target changes to `backend.agents.cognition.perceive.perceive`.

**How to avoid:** After the refactor, verify that existing `test_simulation.py` tests still pass. If they were patching `backend.simulation.engine.perceive`, they will need to be updated to patch `backend.agents.cognition.perceive.perceive` instead. Check test_simulation.py for any such patches before writing new tests.

**Warning signs:** Tests that passed before the ARCH-02 change start failing with `AssertionError: Expected call not found`.

---

### Pitfall 5: Event Tick Advancement Timing

**What goes wrong:** Calling `ev.tick(self._tick_count)` before the agent steps means an event created on tick N expires on tick `N + EVENT_EXPIRY_TICKS` — but if you call tick AFTER incrementing `_tick_count`, the timing shifts by one.

**Why it happens:** `_tick_count` is incremented after agent steps (line 235: `self._tick_count += 1`). If expiry advancement runs after the increment, the event has one fewer effective tick than if it runs before.

**How to avoid:** Advance event lifecycle and clean up expired events AFTER `_tick_count += 1`. This means an event created on tick 0 with `expires_after_ticks=10` is live for ticks 1-10 and cleaned up when tick becomes 10 (10-0 >= 10 → True). This is consistent with the existing `is_expired()` unit tests (test_events.py::TestEventExpiry).

---

### Pitfall 6: `Agent.decide()` kwargs — `config` is no longer needed in the call

**What goes wrong:** The current `decide_action()` call passes `agent_scratch=config.scratch, agent_spatial=config.spatial` explicitly (using `config = agent.config`). After switching to `agent.decide()`, these are read from `self.config` inside the wrapper — the `config` local variable is still needed for `_extract_known_locations(config.spatial.tree)` (line 400), but not for the decide call.

**Why it happens:** `Agent.decide()` closes over `self.config.scratch` and `self.config.spatial` — they must not be passed again via kwargs or they'll conflict.

**How to avoid:** The `config = agent.config` local variable on line 293 can remain for `_extract_known_locations(config.spatial.tree)`. Just don't pass `agent_scratch`/`agent_spatial` as kwargs to `agent.decide()` — the wrapper already handles them.

## Code Examples

### inject_event() with Event lifecycle (complete)
```python
# Source: backend/schemas/events.py + engine.py (verified)
from backend.schemas.events import Event, EVENT_EXPIRY_TICKS
import uuid

async def inject_event(self, text: str, mode: str, target: str | None = None) -> None:
    text = text[:500]

    if mode == "broadcast":
        targets = list(self._agents.keys())
    elif mode == "whisper" and target and target in self._agents:
        targets = [target]
    else:
        logger.warning("inject_event: invalid mode=%s or unknown target=%s", mode, target)
        return

    # EVTS-01: Create Event object with active status
    event_id = str(uuid.uuid4())
    event = Event(
        text=text,
        mode=mode,
        target=target,
        status="created",
        created_tick=self._tick_count,
        expires_after_ticks=EVENT_EXPIRY_TICKS,
    )
    event.tick(self._tick_count)  # transitions created -> active immediately
    self._active_events[event_id] = event

    for agent_name in targets:
        await add_memory(
            simulation_id=self.simulation_id,
            agent_id=agent_name,
            content=f"Event: {text}",
            memory_type="event",
            importance=8,
        )
        agent_obj = self._agents.get(agent_name)
        if agent_obj:
            agent_obj.path = []
```
