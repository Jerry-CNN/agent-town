# Phase 09.1: Code Review Report

**Reviewed:** 2026-04-11
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Three files were reviewed: the core simulation engine, integration tests for event lifecycle, and unit tests for engine concurrency and WebSocket wiring. No critical security or crash-level issues were found. The most significant finding is a logic bug (WR-02) in `_agent_step` that causes every whisper event to record every agent in `heard_by`, regardless of the whisper's intended target — this violates the EVTS-02 requirement and the D-09 design note. A secondary bug (WR-01) causes an incorrect timeout value to be logged on timeout errors. The test suite has a structural gap (WR-04/WR-05) where the event expiry test bypasses the engine entirely and no negative-case assertion covers the WR-02 bug.

## Warnings

### WR-01: Timeout error log reports wrong timeout value

**File:** `backend/simulation/engine.py:281`
**Issue:** In `_agent_step_safe`, the `except asyncio.TimeoutError` block recomputes `timeout` by calling `self.tick_interval` again. `tick_interval` calls `get_adaptive_tick_interval()` which reads a rolling average that may have changed since the timeout was set on line 275. The logged value can differ from the actual timeout that fired, making the warning message misleading for debugging.

**Fix:**
```python
async def _agent_step_safe(self, agent_name: str, agent: Agent) -> None:
    try:
        timeout = max(self.tick_interval * 2, 120)  # computed once
        await asyncio.wait_for(
            self._agent_step(agent_name, agent),
            timeout=timeout,
        )
    except asyncio.TimeoutError:
        # Reuse the same timeout variable — do NOT recompute self.tick_interval here
        logger.warning(
            "Agent %s step timed out after %.0fs — skipping tick",
            agent_name,
            timeout,
        )
    except Exception as exc:
        logger.warning("Agent %s step failed: %s", agent_name, exc, exc_info=True)
```

### WR-02: EVTS-02 — whisper heard_by tracks ALL agents, not just the target

**File:** `backend/simulation/engine.py:322-326`
**Issue:** The EVTS-02 implementation appends `agent_name` to `ev.heard_by` for every agent that runs `_agent_step`, without checking whether `ev.target == agent_name`. A whisper directed at "Alice" will have "Bob", "Carol", and every other agent appended to `heard_by`. This contradicts the EVTS-02 requirement ("heard_by is updated when an agent perceives a whisper event") and the design note D-09 ("Broadcasts do not track propagation"). Whispers are meant to be private; `heard_by` should only include the intended recipient.

```python
# Current code (broken):
for ev in self._active_events.values():
    if ev.mode == "whisper" and not ev.is_expired(self._tick_count):
        if agent_name not in ev.heard_by:
            ev.heard_by.append(agent_name)
```

**Fix:** Add the target guard:
```python
for ev in self._active_events.values():
    if ev.mode == "whisper" and not ev.is_expired(self._tick_count):
        if ev.target == agent_name and agent_name not in ev.heard_by:
            ev.heard_by.append(agent_name)
            logger.debug("[%s] heard whisper event '%s...'", agent_name, ev.text[:30])
```

### WR-03: `_last_schedule_block` set as undeclared dynamic attribute on a dataclass

**File:** `backend/simulation/engine.py:420`
**Issue:** `agent._last_schedule_block = current_schedule_block` assigns an attribute that is not declared in `Agent`'s `@dataclass` definition. The `# type: ignore[attr-defined]` suppressor hides the type error rather than fixing it. Because `Agent` is a plain (non-frozen) dataclass this works at runtime, but it is fragile: if `Agent` is ever marked `frozen=True`, or if static analysis tooling is tightened, this silently breaks and the gating logic stops working.

**Fix:** Add the field to the `Agent` dataclass with a `None` default:
```python
# In backend/agents/agent.py, inside the Agent dataclass:
_last_schedule_block: str | None = field(default=None, repr=False)
```
Then remove the `# type: ignore` comment in `engine.py`.

### WR-04: `test_expired_events_removed_after_tick` tests inline logic, not the engine

**File:** `tests/test_runtime_events.py:196-228`
**Issue:** The test manually replicates the expiry loop from `_tick_loop` inline (lines 221-224) rather than calling the engine method that actually contains that logic. This means the test passes regardless of whether the engine's `_tick_loop` has bugs in expiry handling — it only validates the test's own code. If the expiry condition or dictionary mutation order in `_tick_loop` ever changes, this test will not catch it.

**Fix:** Either (a) extract the expiry purge into a named helper method `_purge_expired_events(tick)` and call it from both the test and `_tick_loop`, or (b) run a minimal tick via the engine:
```python
# Option A: extract helper in engine.py
def _purge_expired_events(self) -> None:
    for eid, ev in list(self._active_events.items()):
        ev.tick(self._tick_count)
        if ev.is_expired(self._tick_count):
            del self._active_events[eid]

# Then call in _tick_loop:
self._tick_count += 1
self._purge_expired_events()

# And in the test:
engine._tick_count = EVENT_EXPIRY_TICKS - 1
engine._tick_count += 1
engine._purge_expired_events()
assert len(engine._active_events) == 0
```

### WR-05: EVTS-02 test has no negative assertion — WR-02 bug goes undetected

**File:** `tests/test_runtime_events.py:137-188`
**Issue:** `test_heard_by_updated_on_whisper_perception` verifies that "Alice" appears in `heard_by` for a whisper targeted at Alice, but it does not assert that a second agent (e.g. "Bob") would NOT be added to `heard_by` for a whisper not targeting Bob. Without this negative assertion, the WR-02 bug (all agents are added regardless of target) passes this test undetected. The test also does not verify the target-matching guard at all.

**Fix:** Add a second agent and a negative assertion:
```python
# Add Bob to the engine
cfg_bob = _make_agent_config("Bob", (7, 7))
engine._agents["Bob"] = Agent(
    name="Bob", config=cfg_bob, coord=(7, 7),
    path=[], current_activity="idle", schedule=[],
)

# After running _agent_step for Alice:
assert "Alice" in whisper_event.heard_by
assert "Bob" not in whisper_event.heard_by, (
    "Bob should not be in heard_by for a whisper targeting Alice"
)
```

### IN-01: `current_tick` variable name misleads — it holds interval duration, not tick number

**File:** `backend/simulation/engine.py:248`
**Issue:** `current_tick = self.tick_interval` assigns a duration (seconds) to a variable named `current_tick`, which readers naturally interpret as the current tick counter integer. The actual tick counter is `self._tick_count`. This naming confusion appears in both the log line and the broadcast payload.

**Fix:** Rename to `tick_interval_sec` or `next_interval`:
```python
next_interval = self.tick_interval
logger.info("===== Tick %d complete | next in %.1fs =====", self._tick_count, next_interval)
if self._broadcast_callback is not None:
    await self._broadcast_callback({
        "type": "tick_interval_update",
        "tick_interval": round(next_interval, 1),
    })
await asyncio.sleep(next_interval)
```

### IN-02: Monkey-patching `_agent_step` as instance attribute relies on implicit behavior

**File:** `tests/test_simulation.py:104`
**Issue:** `engine._agent_step = fake_agent_step` sets an unbound function on the instance. `_agent_step_safe` calls `await self._agent_step(agent_name, agent)` — because instance attribute lookup precedes class method lookup, `self` is NOT prepended. `fake_agent_step(agent_name, state)` receives 2 args as expected. This works, but only because `_agent_step_safe` calls the method via `self._agent_step(...)` (not `SimulationEngine._agent_step(self, ...)`). If the call site ever changes, the tests break silently with a wrong-arg-count TypeError rather than a meaningful failure.

**Fix:** Use `unittest.mock.patch.object` for clarity:
```python
with patch.object(engine, "_agent_step", side_effect=fake_agent_step):
    async with asyncio.TaskGroup() as tg:
        ...
```

### IN-03: Manual patch context manager expansion is fragile

**File:** `tests/test_simulation.py:581`
**Issue:** `with patches[0], patches[1], patches[2], patches[3], patches[4], patches[5], patches[6]:` expands the list by index. If `_make_lifespan_patches` ever adds or removes a patch, callers need manual index updates. This pattern appears in three tests.

**Fix:** Use `contextlib.ExitStack`:
```python
import contextlib
with contextlib.ExitStack() as stack:
    for p in patches:
        stack.enter_context(p)
    with TestClient(app) as client:
        ...
```

### IN-04: `inject_event` silently discards a whisper when `target` is unknown

**File:** `backend/simulation/engine.py:616-620`
**Issue:** When `mode="whisper"` but the `target` name is not in `self._agents`, the method logs a warning and returns without creating any Event object. This is documented in the docstring, but the caller (`ws.py` upstream) receives no indication of the rejection — the `await engine.inject_event(...)` call returns `None` in both success and failure paths. A client that sends a whisper to a misspelled agent name gets no feedback.

**Fix:** This is an info-level concern since validation is documented. Consider raising a `ValueError` or returning a status enum so the WebSocket handler can reply to the client with an error payload. No change required to fix a bug, but worth tracking for UX.
