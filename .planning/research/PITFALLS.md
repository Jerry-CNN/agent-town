# Pitfalls Research

**Domain:** Pixel-art UI upgrade — porting CuteRPG tilesets from Phaser to PixiJS v8, Tiled map integration, animated agent sprites in @pixi/react v8
**Researched:** 2026-04-11
**Confidence:** HIGH (drawn from codebase analysis of reference repo + reference Phaser code + PixiJS v8 official docs + verified community sources)

This file covers pitfalls specific to the v1.2 milestone: porting the CuteRPG tileset assets (18 sheets) from a Phaser 3 reference implementation to PixiJS v8, loading a 17-layer Tiled JSON map (140×100 grid), replacing the current Graphics-based `TileMap.tsx` with sprite-based rendering, adding animated agent sprites (4-direction walk cycles), and keeping @pixi/react v8 as the React bridge. The v1.1 pitfalls (OOP refactoring, LLM optimization, reflection system) are already documented and addressed.

## Critical Pitfalls

### Pitfall 1: Phaser Atlas JSON Format Is Incompatible With PixiJS — Direct Copy Silently Produces Empty Sprites

**What goes wrong:**
The reference repo's `sprite.json` is generated by Atlas Packer Phaser (Gamma V2) and uses Phaser's JSONArray atlas format: `frames` is an **array** of objects, each with a `filename` property and a `frame` object with `{w, h, x, y}` key order. PixiJS `Assets.load()` requires `frames` to be a **keyed object** (dictionary) where frame names are top-level keys, each value has `{frame: {x, y, w, h}}` (note different key order), plus `sourceSize`, `spriteSourceSize`, an `animations` dictionary, and a `meta.image` field pointing to the PNG.

If `sprite.json` is copied directly from the reference repo and loaded with `Assets.load()`, PixiJS will not error out — it will parse the JSON as an unknown format and return an empty spritesheet. Sprites created from it will render as invisible objects. The failure is silent: no console error, sprites just have zero dimensions.

The specific incompatibilities in `sprite.json`:
- `frames` is an array (PixiJS requires object)
- Frame entries use `filename` property (PixiJS requires the name as the dictionary key)
- No `animations` section (PixiJS AnimatedSprite requires this to use `sheet.animations.walk_down`)
- No `meta.image` field (PixiJS loader uses this to find the paired PNG)
- `frame` has keys in order `{w, h, x, y}` (PixiJS parser expects `{x, y, w, h}`)

**Why it happens:**
Both Phaser and PixiJS use "texture atlas JSON" but they are different dialects. The reference repo's `sprite.json` was created by a Phaser-specific packer tool. The visual similarity of the JSON structure masks the semantic incompatibility. Developers assume "JSON atlas" means the same thing across both frameworks.

**How to avoid:**
Do not copy `sprite.json` from the reference repo. Write a new PixiJS-format spritesheet JSON manually — it is only 20 frames, so this takes under 10 minutes. The correct PixiJS format for agent sprites:

```json
{
  "frames": {
    "down-walk.000": { "frame": { "x": 0, "y": 0, "w": 32, "h": 32 }, "sourceSize": { "w": 32, "h": 32 } },
    "down-walk.001": { "frame": { "x": 32, "y": 0, "w": 32, "h": 32 }, "sourceSize": { "w": 32, "h": 32 } },
    ...
    "up": { "frame": { "x": 32, "y": 96, "w": 32, "h": 32 }, "sourceSize": { "w": 32, "h": 32 } }
  },
  "animations": {
    "down-walk": ["down-walk.000", "down-walk.001", "down-walk.002", "down-walk.003"],
    "left-walk": ["left-walk.000", "left-walk.001", "left-walk.002", "left-walk.003"],
    "right-walk": ["right-walk.000", "right-walk.001", "right-walk.002", "right-walk.003"],
    "up-walk": ["up-walk.000", "up-walk.001", "up-walk.002", "up-walk.003"]
  },
  "meta": {
    "image": "agent_sprite.png",
    "format": "RGBA8888",
    "size": { "w": 96, "h": 128 },
    "scale": 1
  }
}
```

The reference repo's `texture.png` files are `96×128` pixels (confirmed from codebase inspection). The `sprite.json` has 20 frames across 4 rows × 4 walk frames + 1 idle per direction = correct for a 96×128 sheet.

**Warning signs:**
- `AnimatedSprite` renders with zero width/height in PixiJS devtools
- `Assets.load('sprite.json')` returns an object but `sheet.textures` is empty
- No console errors despite sprites being invisible
- `sheet.animations` is `undefined` or `{}`

**Phase to address:** Sprite Animation phase. Write and validate the PixiJS-format spritesheet JSON before implementing `AnimatedSprite` component. Test with a minimal standalone PixiJS script (not inside React) first.

### Pitfall 2: Tileset Image Paths in Tiled JSON Embed Absolute Local Filesystem Paths — They Will Fail in Browser

**What goes wrong:**
The reference repo's `tilemap.json` stores tileset image references as relative paths from the Tiled project file: `"image": "map_assets/cute_rpg_word_VXAce/tilesets/CuteRPG_Field_B.png"` (confirmed from codebase inspection). When Agent Town's new Tiled map is designed and exported, the Tiled editor will embed image paths relative to the `.tmx`/`.tmj` file's directory. If assets are reorganized when copying into `frontend/public/`, these paths break silently — `Assets.load()` attempts to fetch `/map_assets/cute_rpg_word_VXAce/...` from localhost, gets 404, and the tileset renders as blank tiles.

The pixi-tiledmap loader resolves tileset image paths relative to the map JSON's URL. If the map JSON is at `public/assets/map/tilemap.json` and the tilesets are at `public/assets/map/tilesets/`, the Tiled file must store paths as `"image": "tilesets/CuteRPG_Field_B.png"` — matching the exact directory structure under `public/`.

**Why it happens:**
Tiled embeds paths relative to its save location, which is usually a developer's local project root, not the `public/` directory where browser assets live. Reorganizing assets when setting up the frontend project creates a path mismatch that is not caught by any tooling.

**How to avoid:**
Design the Tiled project file (`.tmx`/`.tmj`) **inside** the Vite project's `public/assets/` directory from the start, with tilesets also inside `public/assets/`. Keep all assets co-located with the Tiled file so relative paths stay valid when served. The recommended layout:

```
frontend/public/assets/
  map/
    tilemap.json         ← Tiled export here
    tilesets/
      CuteRPG_Field_B.png
      CuteRPG_Field_C.png
      ...
    agents/
      sprite.json
      agent_sprite.png
```

Tiled's `image` paths in the JSON will then be `"tilesets/CuteRPG_Field_B.png"` — which resolves correctly when loaded from `/assets/map/tilemap.json`.

**Warning signs:**
- Browser console shows 404 errors for PNG files when the map loads
- Tileset loads but tiles render as pink/magenta (PixiJS error texture)
- Working in dev mode fails at `Assets.load()` with network errors for tileset images

**Phase to address:** Tileset Asset Setup phase — establish the directory structure before any Tiled editing or pixi-tiledmap integration code is written.

### Pitfall 3: 18 Tilesets Exceed WebGL's Minimum Texture Unit Guarantee — Requires CompositeTilemap

**What goes wrong:**
The reference repo uses 18 tileset PNG files. WebGL 1 guarantees only 8 texture units (`gl.MAX_TEXTURE_IMAGE_UNITS >= 8`); most desktop GPUs provide 16. `@pixi/tilemap`'s `Tilemap` class can use at most `gl.MAX_TEXTURE_IMAGE_UNITS` textures simultaneously per draw call. With 18 tilesets and a single `Tilemap`, tiles from tilesets 17 and 18 (the `blocks_2` and `blocks_3` sheets) may not render on devices with ≤16 texture units, silently producing blank tiles.

`CompositeTilemap` solves this by splitting tiles across multiple layered `Tilemap` instances, each within the hardware limit. The pixi-tiledmap library uses `CompositeTilemap` internally for this reason.

**Why it happens:**
Developers see the `Tilemap` API (simpler, more direct) and use it directly rather than `CompositeTilemap`. The failure only appears on certain hardware — a developer's GPU may have 32 texture units and never hit the bug, but a user's integrated GPU fails silently.

**How to avoid:**
Use `pixi-tiledmap` (which wraps `CompositeTilemap`) rather than building a manual tileset renderer with `@pixi/tilemap`'s lower-level `Tilemap`. If implementing manually, always use `CompositeTilemap` and never assume more than 8 texture units are available. The `pixi-tiledmap` package handles this automatically.

Note from confirmed data: the reference map has tileset GIDs up to 32325 (tileset 18, `blocks_3`). This means any renderer must handle 18 distinct texture sources simultaneously during rendering of certain tiles.

**Warning signs:**
- Tiles render correctly on developer's machine but appear blank on other machines
- Chrome devtools shows WebGL warnings: "Too many textures" or GL errors on draw
- Collision tiles (tileset 16-18, `blocks` series) render but decoration tiles (tilesets 1-15) do not, or vice versa depending on order

**Phase to address:** Tileset Integration phase. Validate with `CompositeTilemap` or `pixi-tiledmap` before writing any manual Tilemap code.

### Pitfall 4: Tiled GIDs Encode Flip Flags in High Bits — Raw GID Values Used as Tileset Lookups Will Crash

**What goes wrong:**
Tiled stores flip and rotation flags in the high bits of tile GIDs. A tile with GID `2684386685` in the Collisions layer (confirmed from analysis of `tilemap.json`) is actually tile `32125` flipped horizontally and diagonally — the real GID is `2684386685 & ~(0x80000000 | 0x40000000 | 0x20000000)` = `32125`. If the Tiled map data is parsed naively and the raw GID is used to look up which tileset it belongs to (e.g., `rawGID >= tileset.firstgid`), the lookup will fail to find any tileset for values like `2684386685` and either throw an error or render the wrong tile.

This is not hypothetical: the reference map's Collisions layer contains both `32125` (plain) and `2684386685` (same tile, flipped) entries. Any custom parser that does not strip flip flags will mishandle 50% of collision tiles.

**Why it happens:**
The Tiled JSON specification section on GID encoding is easy to miss. Developers implementing a parser copy-paste GID-to-tileset lookup logic without checking the spec's note about high-bit flip flags. The pixi-tiledmap library handles this automatically; manual parsers typically do not.

**How to avoid:**
Always mask GIDs before tileset lookup:
```typescript
const FLIP_MASK = ~(0x80000000 | 0x40000000 | 0x20000000) >>> 0;
const realGID = gid & FLIP_MASK;
const flippedH = !!(gid & 0x80000000);
const flippedV = !!(gid & 0x40000000);
const flippedD = !!(gid & 0x20000000);
```
If using `pixi-tiledmap`, flip flags are handled internally. If writing a custom collision-grid extractor (to build the walkability array for backend pathfinding), include the mask.

This matters specifically for the collision-layer-to-walkability-grid extraction that replaces the current `town.json`-based collision system — when building the walkability array from the Tiled map's "Collisions" layer, raw GIDs must be masked before comparing to `firstgid`.

**Warning signs:**
- Collision tiles appear to render correctly on most tiles but some are blank
- `RangeError: invalid tileset for gid` console errors for specific tiles
- Walkability grid has unexpected holes where flipped collision tiles should be

**Phase to address:** Tileset Integration phase AND Collision Extraction phase (extracting the walkability grid from the new Tiled map to replace `town.json` collision data).

### Pitfall 5: `TextureStyle.defaultOptions.scaleMode` Must Be Set Before Any Texture Is Loaded — Otherwise Pixel Art Looks Blurry

**What goes wrong:**
PixiJS v8 defaults to `scaleMode: 'linear'` (bilinear filtering). Pixel art sprites rendered with linear filtering appear blurry when scaled — especially visible when the map is zoomed out (current `FIXED_SCALE = 0.45` in `MapCanvas.tsx`) because each 32×32 tile is rendered at ~14×14 pixels with interpolation bleeding between pixels. Setting `TextureStyle.defaultOptions.scaleMode = 'nearest'` after any texture has already been loaded has no effect on those textures; only textures loaded after the setting change are affected. In a React app where `Assets.load()` may be called in a `useEffect` early in the component lifecycle, the window to set this before texture creation is narrow.

**Why it happens:**
PixiJS v8 changed the texture configuration API from v7 — in v7 the scale mode was set on `PIXI.settings.SCALE_MODE`. Developers migrating from v7 or following outdated tutorials set the global setting on `PIXI.settings` (which is removed in v8) or set it after the `Assets.load()` call where textures are already cached. The error is silent — the textures load fine, they just have the wrong filter applied permanently.

**How to avoid:**
Set `TextureStyle.defaultOptions.scaleMode = 'nearest'` as the **very first line** of the PixiJS initialization code, before any `Assets.load()` or `Application` creation:

```typescript
// main.tsx or the module that initializes PixiJS
import { TextureStyle } from 'pixi.js';
TextureStyle.defaultOptions.scaleMode = 'nearest'; // MUST be before Assets.load()
```

Also set the canvas CSS property for pixelated rendering (prevents browser-level anti-aliasing):
```css
canvas { image-rendering: pixelated; }
```

And enable round pixels in `Application` init for crisp tile alignment:
```typescript
<Application roundPixels={true} ...>
```

**Warning signs:**
- Tiles look blurry or fuzzy at all zoom levels
- Individual pixels appear as soft gradients instead of hard edges
- The blur is worse on tiles zoomed to non-integer scale factors (e.g., 0.45×)
- `TextureStyle.defaultOptions.scaleMode` is set after a `useEffect` that calls `Assets.load()`

**Phase to address:** Asset Loading Setup phase — the very first phase. Must be the first configuration in `main.tsx` or wherever PixiJS is initialized.

### Pitfall 6: `AnimatedSprite` Is Not in the @pixi/react v8 `extend()` Call — Causes Runtime Error

**What goes wrong:**
`@pixi/react v8` requires every PixiJS class used as a JSX element to be registered via `extend({ ClassName })`. The current `MapCanvas.tsx` calls `extend({ Container, Graphics, Text })`. Adding `AnimatedSprite` JSX usage (`<pixiAnimatedSprite ...>`) without adding `AnimatedSprite` to the `extend()` call throws a runtime error: `Unknown PixiJS component type: pixiAnimatedSprite`. The error only appears at runtime, not during TypeScript compilation, because `@pixi/react` dynamically resolves JSX element names.

Additionally, `Sprite` itself (needed if using static idle frames between walk cycles) must also be added to `extend()`. Missing either results in the same runtime error.

**Why it happens:**
The `extend()` call is typically written once at app initialization and not updated when new PixiJS components are added. The TypeScript type system does not catch missing `extend()` registrations because `@pixi/react v8` uses string-based JSX name resolution at runtime.

**How to avoid:**
Update `MapCanvas.tsx`'s `extend()` call to include all PixiJS classes used anywhere in the app:
```typescript
import { Container, Graphics, Text, AnimatedSprite, Sprite } from 'pixi.js';
extend({ Container, Graphics, Text, AnimatedSprite, Sprite });
```

As a rule: every time a new PixiJS component is added to any component in the tree, add it to this `extend()` call. Consider centralizing `extend()` in `main.tsx` and calling it once for the entire app to avoid maintenance drift.

**Warning signs:**
- Runtime error: `Unknown PixiJS component type: pixi[ClassName]`
- Error appears in the browser console only after the component renders, not during build
- TypeScript compilation succeeds but the app crashes in the browser
- Only the new component fails; previously registered components render correctly

**Phase to address:** Sprite Animation phase — update `extend()` before writing any `AnimatedSprite` component code.

### Pitfall 7: The `useTick` Callback Is Not Memoized — Re-Registration Every Frame Causes Memory Pressure With 25 Agents

**What goes wrong:**
The current `AgentSprite.tsx` already uses `useTick` correctly with stable callbacks. However, when `AgentSpriteInner` is refactored to use `AnimatedSprite`, the animation direction switching logic (`if (movingLeft) play('left-walk') else if (movingRight) play('right-walk')...`) is typically written inline in the `useTick` callback body. If this callback captures props or state (e.g., `agentId`, or a local `direction` state variable), and is not wrapped in `useCallback`, `useTick` de-registers and re-registers the callback on every frame for every agent.

At 25 agents running at 60 FPS, this is 1,500 callback churn events per second in the PixiJS ticker. Each churn event is a de-register + re-register operation on the shared ticker's listener array. This is confirmed as a known @pixi/react issue: "the callback passed to useTick is not memoised, which can cause issues where your callback is being removed and added back to the ticker on every frame."

**Why it happens:**
`useTick` does not internally memoize its callback argument. The natural refactoring pattern of extracting direction logic into an inline function inside `useTick` creates a new function object every render, triggering the de-registration cycle.

**How to avoid:**
All `useTick` callbacks must be wrapped in `useCallback`. The current `AgentSprite.tsx` pattern (using refs instead of state for animation control, reading from `useSimulationStore.getState()` instead of hooks inside the callback) is already the correct approach. Extend this pattern to animation direction switching:

```typescript
const tickCallback = useCallback(() => {
  const agent = useSimulationStore.getState().agents[agentId];
  if (!agent || !spriteRef.current) return;
  // direction logic using refs only — no state captures
  const dx = target.x - cur.x;
  const dy = target.y - cur.y;
  const dir = resolveDirection(dx, dy);
  if (dirRef.current !== dir) {
    dirRef.current = dir;
    spriteRef.current.textures = sheet.animations[`${dir}-walk`];
    spriteRef.current.play();
  }
}, [agentId]); // agentId is stable — never changes for a given agent
```

The key constraint: `agentId` is the only dependency; all other data is read from Zustand `getState()` (not hooks) or from refs.

**Warning signs:**
- Browser memory usage grows over time with agents walking
- Chrome devtools performance tab shows ticker listener array growing
- FPS drops proportionally to agent count (rather than the expected constant overhead)
- `console.log` inside `useTick` fires twice per frame for some agents

**Phase to address:** Sprite Animation phase — validate ticker callback registration count before shipping the animated sprite component.

### Pitfall 8: Non-Visual Tiled Layers (Sector Blocks, Arena Blocks, Collision, Spawning) Must Not Be Rendered

**What goes wrong:**
The reference map has 17 layers. Ten are visual (Bottom Ground through Foreground L2). Seven are metadata: Collisions, Object Interaction Blocks, Arena Blocks, Sector Blocks, World Blocks, Spawning Blocks, Special Blocks Registry. These metadata layers use colored blocks from `blocks_1.png`, `blocks_2.png`, `blocks_3.png` tilesets to encode navigation zones — each block color represents a different sector or arena. They are not intended for display.

If `pixi-tiledmap` renders all 17 layers (its default behavior), the colored navigation blocks from Sector Blocks, Arena Blocks, etc. appear on top of the visible map, obscuring the town with solid-colored rectangles.

**Why it happens:**
`pixi-tiledmap` renders every layer by default. The navigation layers look like normal tile layers to the renderer — they use tilelayer type like all others. The Phaser reference implementation explicitly skips these layers in its render code (only `createLayer()` is called for the visual layers; the collision/block layers are created and then `setDepth(-1)` is applied or simply not added to visible groups).

**How to avoid:**
When constructing the renderer, explicitly control which layers are rendered visually. Either:
1. Use pixi-tiledmap's layer filtering: render only layers matching a name whitelist (`["Bottom Ground", "Exterior Ground", ..., "Foreground L2"]`)
2. Set the `visible: false` flag on the non-visual layers after loading (pixi-tiledmap respects Tiled's `visible` property — set the metadata layers to invisible in the Tiled editor itself before exporting)

Option 2 is more maintainable: open the new Tiled map, select each metadata layer (Collisions, Sector Blocks, Arena Blocks, etc.), and uncheck "Layer > Visible" before exporting. Then `pixi-tiledmap` honors these flags automatically.

The Collisions layer is additionally useful for building the walkability grid (backend path-finding), but that extraction is done in JavaScript from the raw JSON data, not from the rendered tilemap.

**Warning signs:**
- Map renders but has solid colored rectangles overlaying the town
- Specific areas (sector boundaries) appear as colored blocks instead of tiles
- The overhead view matches the sector boundary layout from town.json

**Phase to address:** Tiled Map Integration phase — set invisible flags in Tiled editor for all seven metadata layers before exporting the JSON.

### Pitfall 9: Foreground Layers Must Render Above Agent Sprites — Depth Ordering Is Not Automatic

**What goes wrong:**
The reference Phaser implementation explicitly sets `foregroundL1Layer.setDepth(2)` and `foregroundL2Layer.setDepth(2)` so that trees, building overhangs, and roof edges appear above agents (creating the "walking behind" effect for indoors/under trees). In PixiJS with @pixi/react, `zIndex` controls rendering order within a container, but there is no automatic depth sorting between the tilemap container and agent sprite containers.

If agents are added to the same container as the tilemap or as siblings at the same level, they will render either always above or always below all map layers, breaking the depth illusion. Agents should render above ground/decoration layers but below foreground layers.

**Why it happens:**
In React-based PixiJS apps, the natural structure is `<TileMap/>` followed by `<AgentSprite/>` components inside the same `pixiContainer`. This means agents always render above all tilemap layers, including the foreground. The Phaser depth API has no direct equivalent in @pixi/react's JSX structure.

**How to avoid:**
Structure the scene graph to explicitly interleave map layers and agents at the right depth. When using `pixi-tiledmap`, the loaded `container` exposes each layer. The rendering order in @pixi/react must be:

```
MapContainer
  ├── BottomGroundLayer (z=0)
  ├── ExteriorGroundLayer (z=1)
  ├── ExteriorDecorationL1Layer (z=2)
  ├── ExteriorDecorationL2Layer (z=3)
  ├── InteriorGroundLayer (z=4)
  ├── WallLayer (z=5)
  ├── InteriorFurnitureL1Layer (z=6)
  ├── InteriorFurnitureL2Layer (z=7)
  ├── AgentSpritesContainer (z=8)  ← agents render here
  ├── ForegroundL1Layer (z=9)      ← foreground above agents
  └── ForegroundL2Layer (z=10)
```

With `pixi-tiledmap`, access individual layer containers via `tiledMap.getChildByName('Foreground L1')` and manually reparent them to achieve this ordering. Set `sortableChildren = true` on the parent container and use `zIndex` on each layer container.

**Warning signs:**
- Agents are always visible even when walking through interior doorways or behind trees
- The "walking behind a building" effect is absent
- Agents appear to float above rooftops that should occlude them
- Foreground decoration (e.g., tree tops) appears behind agent sprites

**Phase to address:** Tileset Integration phase — scene graph depth ordering must be designed before the first end-to-end map render. Do not defer depth ordering as a cosmetic fix.

### Pitfall 10: The `pixi-tiledmap` Container Must Be Added to the PixiJS Stage Imperatively — Not as JSX

**What goes wrong:**
`pixi-tiledmap` returns a PixiJS `Container` from `Assets.load()`. This container is a plain PixiJS object, not a React element, and cannot be used as JSX. The common mistake is attempting to render it as `<pixiContainer ref={tiledMapRef}>` and then adding the pixi-tiledmap result as a child imperatively in a `useEffect`. This causes a React render cycle collision: the `useEffect` imperative child-add is overwritten by React's reconciler on the next render, making the tilemap appear and then disappear.

**Why it happens:**
@pixi/react v8 manages the PixiJS scene graph through React's reconciler. When you add children imperatively to a container that React is also managing (via JSX children), React's next reconcile overwrites the imperative additions. This is a documented footgun in @pixi/react.

**How to avoid:**
Use a dedicated container that React does not touch for the tilemap. The correct pattern:

```typescript
function TileMapLayer() {
  const containerRef = useRef<PixiContainer | null>(null);

  useEffect(() => {
    if (!containerRef.current) return;
    let mounted = true;
    Assets.load('/assets/map/tilemap.json').then(({ container }) => {
      if (!mounted || !containerRef.current) return;
      containerRef.current.addChild(container);
    });
    return () => { mounted = false; };
  }, []);

  // pixiContainer with no JSX children — React won't reconcile into it
  return <pixiContainer ref={containerRef} />;
}
```

The `pixiContainer` element has no JSX children, so React's reconciler sees it as a leaf and will not overwrite imperative children added in `useEffect`.

**Warning signs:**
- Tilemap flashes on first render then disappears
- Tilemap re-appears on hot reload but disappears after first state update
- Adding any Zustand subscription to the parent component causes the tilemap to vanish

**Phase to address:** Tileset Integration phase. Establish this pattern in the first integration spike before building out layer management.

### Pitfall 11: Agent Sprites Positioned at Tile Grid Coordinates, Not Pixel Coordinates — Off-By-One on the New 140×100 Map

**What goes wrong:**
The current `AgentSprite.tsx` receives positions from Zustand as `{ x, y }` which are **tile coordinates** (0–99), and it renders them as pixel positions directly: `containerRef.current.x = agent.position.x`. This worked for the current 100×100 map where the backend sends tile coords and the frontend multiplies by `TILE_SIZE` implicitly through the existing TileMap scale. When the map changes to 140×100 tiles, this coordinate transform must be explicit.

More specifically: the backend's `coord` system uses `[col, row]` tile indices. The frontend must convert to pixels as `x = col * TILE_SIZE`, `y = row * TILE_SIZE`. If the current code happens to work because `TILE_SIZE = 32` is embedded correctly, any change to tile size or map origin will break agent positioning. Additionally, the Tiled map is 140×100 tiles = 4480×3200 pixels (vs. the current 3200×3200), so `MAP_SIZE_PX = 3200` in `MapCanvas.tsx` must be updated to account for the non-square map.

**Why it happens:**
The coordinate conversion is implicit in the current implementation. When the map dimensions change and tile-to-pixel conversion is not explicit, agents appear at the correct relative positions but on the wrong absolute pixels, or walk off the map boundaries.

**How to avoid:**
Make the tile-to-pixel conversion explicit in a single utility function used everywhere:
```typescript
const TILE_SIZE = 32;
export function tileToPixel(tileX: number, tileY: number) {
  return { x: tileX * TILE_SIZE, y: tileY * TILE_SIZE };
}
```
Update `AgentSprite.tsx`'s lerp target calculation to always go through `tileToPixel()`. Update `MapCanvas.tsx`'s `MAP_SIZE_PX` to `MAP_WIDTH_TILES * TILE_SIZE` = 4480 (width) and `MAP_HEIGHT_TILES * TILE_SIZE` = 3200 (height). The centering logic must account for the now-non-square map.

**Warning signs:**
- Agents appear in the upper-left quadrant of the map (x coordinate correct, y off by factor of 1.4)
- Agents cluster at the map's 100-tile boundary instead of spreading across all 140 columns
- Map canvas is centered on the old 3200×3200 size but the map extends to 4480×3200

**Phase to address:** Map Layout Redesign phase — update all coordinate constants before integrating new map data.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Copy Phaser's `sprite.json` and patch it in code | Skip writing new JSON format | Brittle: any JSON mutation in code is harder to maintain than a correct source file; re-copy from reference breaks things | Never — write the PixiJS-format JSON once correctly |
| Render all 17 Tiled layers including metadata layers | Simpler initial code (load all layers, don't filter) | Colored navigation blocks overlay the town; debugging is confusing | Never — visibility flags in the Tiled editor are the correct fix |
| Use `Tilemap` instead of `CompositeTilemap` | Simpler API | Silent failure on GPUs with ≤16 texture units for the 18-tileset map | Never — GPUs with 16 texture units are common; use pixi-tiledmap which handles this |
| Hardcode `MAP_SIZE_PX = 3200` after switching to 140×100 map | No change required | Off-by-one in map centering; agents clip at wrong boundary | Never — derive from `MAP_WIDTH_TILES * TILE_SIZE` |
| Skip depth ordering for now, fix later | Faster to ship | Agents visible through building roofs; cosmetically wrong in demo; fixing later requires scene graph refactor | Only acceptable if the milestone explicitly defers "foreground occlusion" as v1.3 scope |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Phaser `sprite.json` → PixiJS `Assets.load()` | Copy `sprite.json` directly from reference repo | Write new PixiJS-format JSON with keyed `frames` object, `animations` section, and `meta.image` |
| `pixi-tiledmap` + `@pixi/react` | Render tilemap container as JSX child | Keep tilemap in a `pixiContainer` with no JSX children; add imperatively in `useEffect` |
| Tiled metadata layers + pixi-tiledmap | Render all layers (default) | Set `visible: false` on metadata layers in Tiled editor before export, or whitelist visual layers |
| 18 tilesets + WebGL hardware limit | Use `Tilemap` directly | Use `pixi-tiledmap` (wraps `CompositeTilemap`) to stay within per-draw-call texture unit limits |
| Tiled GIDs with flip flags | Use raw GID values for tileset lookup | Always apply `gid & ~(0x80000000 | 0x40000000 | 0x20000000)` before tileset resolution |
| Pixel art scale mode | Call `TextureStyle.defaultOptions.scaleMode = 'nearest'` after `Assets.load()` | Set before any `Assets.load()` call, as the first line of PixiJS initialization |
| `@pixi/react extend()` + `AnimatedSprite` | Use `<pixiAnimatedSprite>` without registering class | Add `AnimatedSprite` (and `Sprite`) to the `extend()` call in `MapCanvas.tsx` |
| `useTick` + animation direction logic | Inline direction function captures state → non-memoized callback | Wrap in `useCallback`; use refs and `getState()` inside, not hook subscriptions |
| Tiled path → `frontend/public/` layout | Edit Tiled files from a different directory than `public/assets/` | Tiled project file lives inside `public/assets/map/` so all relative paths stay valid |
| 140×100 map dimensions | Reuse `MAP_SIZE_PX = 3200` from square 100×100 map | Derive separately: `MAP_WIDTH_PX = 140 * 32 = 4480`, `MAP_HEIGHT_PX = 100 * 32 = 3200` |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| `AnimatedSprite` + non-memoized `useTick` callback | Memory growth; FPS drops with more agents; ticker listener array grows | `useCallback` with stable deps; read store via `getState()` not hooks | Immediately visible above 10 agents at 60 FPS |
| All 17 map layers rendered as graphics | High GPU memory from 17 separate draw calls for metadata layers | Invisible metadata layers in Tiled editor; pixi-tiledmap skips them | Initial render shows colored blocks; map re-render on resize is expensive |
| `drawMap` Graphics callback re-run on any Zustand update | Canvas re-renders every WebSocket tick; tile map flickers at 0.5s intervals | Preserve empty `[]` deps array; no Zustand store reads inside the draw callback | Immediately on first store subscription added to `drawMap` callback |
| 18 tilesets × 17 layers = 306 potential texture references | Slow initial load; excessive GPU memory | Use `pixi-tiledmap` batching; load tilesets once via `Assets.add()` before loading the map | On mobile or low-VRAM devices; >2s load time on 10Mbps connection |
| Large interior tileset PNGs (interiors_pt1.png = 512×10016) | GPU upload stall on first render; black flash | Pre-load tilesets in background before showing map; use PixiJS Assets bundle manifest | On first map render; noticeable on slower connections |
